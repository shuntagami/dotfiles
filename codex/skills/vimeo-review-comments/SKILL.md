# Vimeo Review Comments

Vimeo Reviewページが使用する非公開の `private_comments` リクエストを、現在のブラウザセッション内だけで再利用する。認証情報を読み出したり保存したりしない。

## 必須ルール

- Chromeの既存ログイン状態を使う。Cookie、Authorizationヘッダー、guest user IDを表示・コピー・保存しない。
- 投稿は第三者への通信に当たる。全コメントの投稿先、件数、本文を準備してから、最初の `Post` クリック直前にユーザー確認を取る。
- `Post` は一度だけクリックする。クリック後は状態を監視し、失敗表示を見ても再クリックしない。
- 投稿前にページを再読み込みし、過去に注入した `fetch` ラッパーとフォームの失敗状態を消す。
- 既存コメントの取得に失敗した場合は投稿しない。重複確認を省略しない。
- 既存コメントの削除や編集は、このSkillの一括投稿に含めない。必要なら別途確認を取る。
- このエンドポイントは非公開仕様である。リクエスト形状が変わった場合は推測で送信せず、1件の正規UI操作をNetworkで再調査する。

## 1. 入力を正規化する

レビューURLとコメント一覧を一時ファイルに保存し、同梱スクリプトで検証済みのブラウザ用スニペットを生成する。

`SKILL_DIR` を、この `SKILL.md` があるディレクトリの絶対パスにする。

```bash
node "${SKILL_DIR}/scripts/build_batch.mjs" \
  --url '<VIMEO_REVIEW_URL>' \
  --input '<COMMENTS_TXT>' \
  --output /tmp/vimeo-review-comments.js
```

入力形式:

```text
00:04 画像
10:48 テロップを修正
01:02:03 音量を下げる
```

スクリプトは次を行う。

- Review IDとVideo IDをURLから抽出する。
- タイムコードを秒へ変換する。
- 同一タイムコード・同一本文を重複排除する。
- 秒順へソートする。
- 7.5秒間隔、429時の `Retry-After` 待機、最大8回の再試行を設定する。

生成結果の件数と正規化一覧を確認する。動画尺を超える時刻がないこともプレイヤー表示で確認する。

## 2. ブラウザを準備する

1. `chrome:control-chrome` が利用可能なら、そのSkillを読み、対象Chromeタブのページコンテキストでスニペットを評価する。
2. Chrome連携が利用できなければ `computer-use:computer-use` を読み、Chrome DevTools Consoleを使う。
3. 対象Review URLを開き、コメント一覧と入力欄が読み込まれるまで待つ。
4. ログインまたはレビュー用の名前入力が必要なら、ユーザーに完了してもらう。
5. ページを一度再読み込みする。

DevToolsへ入力する場合:

1. Consoleプロンプトを選ぶ。
2. `Cmd+A` で残っている式を必ず全選択する。
3. `/tmp/vimeo-review-comments.js` の内容をそのまま入力する。手作業で編集しない。
4. `Return` を押す。
5. `vimeo-review-comments armed` と件数が表示されることを確認する。

## 3. 一括投稿を開始する

1. Reviewページのコメント欄へ、一覧の先頭コメント本文を入力する。表示中のプレイヤー時刻は任意でよい。スニペットが正しい時刻へ置き換える。
2. ユーザーへ、対象動画、投稿件数、本文一覧を示し、投稿実行の確認を取る。
3. 確認後に `Post` を一度だけクリックする。
4. UIが失敗表示になっても再クリックしない。スニペットの状態を確認する。

スニペットは、本文を含む正規のPOSTだけを捕捉する。空本文や解析不能なPOSTでは作動せず、`armed` のまま待機する。

## 4. 進行を監視する

ChromeのページコンテキストまたはDevTools Consoleで次を評価する。

```js
JSON.stringify(window.__vimeoReviewCommentsBatch?.summary())
```

主な状態:

- `armed`: 正規POST待ち。投稿操作をまだ捕捉していない。
- `checking`: 既存コメントを取得し、重複を照合中。
- `running`: キューを送信中。
- `complete`: 全件成功または既存のためスキップ済み。
- `failed`: 安全に停止。`error` と `results` を確認する。

60秒以上処理が続く場合は、ユーザーへ完了件数と残件数を伝える。

## 5. 完了を検証する

`summary()` で次を確認する。

- `status` が `complete`
- `failed` が `0`
- `pending` が `0`
- 各送信結果がHTTP `201`
- `posted + skippedExisting` が入力件数と一致

完了後にページを再読み込みし、コメントサイドバーで先頭・末尾・途中の複数時刻を照合する。仮想スクロールのため全件が同時にDOMへ出ないことがあるので、HTTP結果を主、画面表示を補助検証とする。

## 障害時の扱い

- `429`: スニペットに任せる。`Retry-After` がなければ60秒待って再試行する。
- `401` / `403`: 投稿を止め、対象ChromeでのログインまたはReview権限を確認する。
- `existing-comments-check-failed`: 投稿を止める。ページ再読み込み後にスニペットを再設置する。
- `armed` のまま: コメント欄を一度編集してから、確認済みなら `Post` を一度だけ押す。連打しない。
- `failed`: 成功済みのタイムコードを `results` から抽出し、新しい入力から除外して再生成する。ページ再読み込み前に状態を記録する。

## 完了報告

次を簡潔に報告する。

- 指定件数、投稿件数、既存のためスキップした件数
- 全送信のHTTP結果
- 最終画面照合の結果
- 既存の不要コメントが残っている場合はその事実
- 非公開APIのため将来変更される可能性
